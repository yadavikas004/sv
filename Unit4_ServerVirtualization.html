vMotion — Live VM Migration

Requirements · Process · CPU Masking · Enhanced vMotion Compatibility

What is vMotion?
🔴 Must Know
vMotion enables live migration of a running VM from one ESXi host to another with zero downtime — users and applications experience no interruption. The VM's memory state, network connections, and storage are transferred transparently.
/* vMotion Process — Step by Step */ Step 1: Shadow VM created on destination host ↓ Step 2: VM memory pages copied to destination (iterative) ↓ Step 3: Memory pages modified during copy → added to dirty-page bitmap ↓ Step 4: Source VM quiesced for a brief moment (stun) ↓ Step 5: Remaining dirty pages copied (fast — very few left) ↓ Step 6: VM resumes on destination; source VM removed ↓ Result: Zero downtime — network connections maintained via RARP
💡
Exam Tip: vMotion process is a classic 6-mark question. Cover: shadow VM creation → iterative memory copy → dirty pages → stun → resume on destination. Mention RARP for network notification.
vMotion Requirements
🔴 Must Know

Network Requirements

  • Dedicated VMkernel port for vMotion on both hosts
  • Minimum 1 GbE network (10 GbE recommended)
  • Both hosts on the same L2 network (or routed vMotion)
  • Hosts must be able to ping each other's vMotion IP

Storage Requirements

  • Shared storage accessible by both hosts (SAN/NAS/vSAN)
  • Or use Storage vMotion to migrate disk simultaneously
  • VM cannot be on local storage of source host (unless Storage vMotion)

CPU Requirements

  • CPUs must be compatible (same vendor: Intel or AMD)
  • CPU features exposed to VM must be available on destination
  • Use CPU Masking or EVC to handle CPU differences

Other Requirements

  • vCenter Server required (manages the migration)
  • Both hosts in same vCenter (or use Cross-vCenter vMotion)
  • VMware Tools installed in VM (recommended)
  • VM must not have CD/DVD connected to local ISO
CPU Masking & Enhanced vMotion Compatibility (EVC)
🔴 Must Know
CPU Masking hides specific CPU feature bits (CPUID) from VMs so they don't know which physical CPU features are available — enables vMotion between hosts with slightly different CPUs of the same family.
EVC (Enhanced vMotion Compatibility) is a cluster-level setting that automatically configures CPU masking for ALL hosts and VMs in a cluster — ensuring vMotion compatibility across all hosts with different CPU steppings.
FeatureCPU MaskingEVC
ScopePer VM (manual)Entire cluster (automatic)
ConfigurationEdit VM settings → CPU → Advanced (CPUID mask)Cluster Settings → EVC → Select CPU baseline
Use caseOne-off migration between different CPU hostsMixed-CPU clusters needing cluster-wide vMotion
CPU vendor crossingCannot cross Intel ↔ AMDCannot cross Intel ↔ AMD
Ease of useComplex — manual CPUID editingSimple — select baseline (e.g., "Intel Nehalem")
💡
Exam Tip: EVC is preferred over per-VM CPU masking in production. Key rule: vMotion CANNOT migrate between Intel and AMD CPUs regardless of masking.

Storage vMotion & Cross-vCenter vMotion

Storage vMotion
🔴 Must Know
Storage vMotion migrates a running VM's virtual disk files from one datastore to another with zero downtime — the VM continues to run while its VMDK is relocated.
FeaturevMotionStorage vMotion
What movesVM compute state (memory, vCPU)VM disk files (VMDKs)
VM stateRunning during migrationRunning during migration
Shared storage needed?Yes (for disk)No — moves FROM one datastore TO another
Use caseLoad balance compute across hostsMove VM from full datastore, storage maintenance
Network disruptionNone (RARP sent)None

Combined vMotion (host + storage simultaneously):

  • Move VM's compute AND storage simultaneously in a single operation
  • Useful when destination host doesn't share storage with source
  • Requires sufficient network bandwidth for both memory and disk transfer
Use case: Storage maintenance, rebalancing, tiering
Changed block tracking: Tracks modified blocks during migration
Cross-vCenter vMotion (vSphere 6.0+)
🟠 High Priority
Cross-vCenter vMotion allows migrating a running VM between ESXi hosts managed by different vCenter Server instances — even across geographic locations (Long Distance vMotion).

Requirements

  • Both vCenters in same SSO domain OR linked mode
  • Shared network between sites (for vMotion traffic)
  • vSphere Enterprise Plus license
  • RTT latency: <150ms for regular, <100ms for Long Distance

Use Cases

  • Data center migration / consolidation
  • Workload mobility between clouds
  • Disaster avoidance (move before disaster)
  • Geographic load balancing
Introduced: vSphere 6.0
Max latency: 150ms RTT
License: Enterprise Plus

Performance Monitoring & Alarms

Overview · Charts · Alarms · Scopes

Performance Monitoring in vSphere
🔴 Must Know
vSphere Performance Monitoring provides real-time and historical metrics for CPU, memory, disk, and network utilization across VMs, ESXi hosts, datastores, and clusters — via the vSphere Web Client's Performance tab.
Metric CategoryKey Metrics to MonitorAlert Threshold
CPUCPU Usage %, CPU Ready, CPU Co-StopCPU Ready >10% = problem
MemoryMemory Usage %, Balloon, Swap Rate, Active MemoryBallooning >0 = memory pressure; Swap >0 = critical
Disk (Storage)Disk Read/Write IOPS, Disk Latency (ms), Disk ThroughputLatency >20ms = concern
NetworkNetwork Transmitted/Received (kbps), Dropped PacketsDropped packets >0 = NIC saturation

Real-Time Charts

20-second interval; last 1 hour of data; shown in Performance → Advanced tab in vSphere Client

Historical Charts

Rolled up data: daily, weekly, monthly, yearly. Useful for capacity planning and trend analysis.

Overview Charts

Quick summary view for host/cluster; shows CPU, memory, disk, network in one panel

Advanced Charts

Custom metric selection, time range, comparison of multiple objects — for deep-dive analysis

💡
Exam Tip: Know the 4 metric categories (CPU, Memory, Disk, Network) and what to look for. CPU Ready >10% = over-provisioned; Memory balloon/swap = memory contention.
Performance Charts in vSphere
🟠 High Priority
Performance Charts are graphical views in the vSphere Web Client that visualize metric data (CPU, memory, disk, network) over time for any inventory object — VMs, hosts, clusters, datastores, or resource pools.
Chart TypeTime RangeCollection IntervalBest Used For
Real-TimeLast 1 hour20 secondsLive troubleshooting — immediate issue detection
Past DayLast 24 hours5 minutes (rolled up)Spotting patterns within a day
Past WeekLast 7 days30 minutes (rolled up)Weekly trend analysis
Past MonthLast 30 days2 hours (rolled up)Monthly capacity planning
Past YearLast 365 days1 day (rolled up)Long-term growth trending

Overview Chart

  • Quick summary view of CPU, memory, disk, network in one panel
  • Available for hosts, VMs, clusters, datastores
  • No customization — preset metrics only
  • Good for quick health check

Advanced Chart

  • Fully customizable — choose any metric, time range, object
  • Compare multiple VMs or hosts on same chart
  • Export chart data to CSV
  • Used for root cause analysis and capacity planning
/* How to Access Performance Charts */ 1. Select object in vSphere Client (VM / Host / Cluster / Datastore) 2. Click "Monitor" tab → "Performance" 3. Choose: Overview (quick) or Advanced (custom) 4. In Advanced: click "Edit Chart Options" → select metrics + time range 5. Use "Export" to download data as CSV for reporting
💡
Exam Tip: Know the 2 chart types: Overview (preset, quick) vs Advanced (custom, any metric). Know the 5 time ranges. Data is "rolled up" at longer intervals — real-time = 20 sec; past year = 1 day average.
Alarms in vSphere
🔴 Must Know
Alarms are automated notifications triggered when specific conditions or thresholds are met in the vSphere environment — they alert administrators and optionally take automatic action.
Alarm ConceptExplanation
Alarm DefinitionConfiguration: trigger condition + action + target object type
TriggerCondition that causes alarm to fire: metric threshold, event, or state change
ActionWhat happens when alarm fires: send email, send SNMP trap, run script, power off VM, etc.
Alarm StateNormal (green) → Warning (yellow) → Alert (red)
AcknowledgmentAdmin acknowledges alarm — stops repeated notifications but alarm remains until condition clears

Alarm Scopes (what objects alarms can monitor):

Host Alarms

CPU usage, memory usage, host connection state, host hardware status (fans, temperature, power)

VM Alarms

CPU/memory usage, VM power state, VMware Tools status, snapshot size

Datastore Alarms

Datastore usage (% full), datastore accessibility, I/O latency

Cluster/Network Alarms

HA/DRS status, vSAN health, distributed switch errors

/* Creating an Alarm — Steps */ 1. In vSphere Client → Select object (host/VM/cluster) 2. Monitor tab → Alarm Definitions → Add 3. Define: Name, Target Type (VM/Host/Datastore etc.) 4. Set Triggers: metric threshold (e.g., CPU > 90% for 5 min) 5. Set Actions: email notification / SNMP trap / run script 6. Set Reporting: frequency of repeated notifications 7. Save → Alarm is active and monitored by vCenter
💡
Exam Tip: "Explain alarms in vSphere" = Definition + Trigger + Action + States (Normal/Warning/Alert) + Scope types. Know the steps to create an alarm. Very likely 6-mark sub-question.

Automating VMware vSphere — PowerCLI

PowerShell · PowerCLI · Installation · Commands

What is PowerCLI?
🔴 Must Know
VMware PowerCLI is a command-line tool built on PowerShell that allows administrators to automate and manage VMware vSphere environments using scripts and cmdlets — enabling bulk operations that would take hours in the GUI.

What you can automate

  • Create, configure, start, stop VMs in bulk
  • Deploy VMs from templates at scale
  • Manage hosts, clusters, networks, storage
  • Generate reports on resource usage
  • Apply patches and updates
  • Manage snapshots across all VMs

Key advantages

  • Automate repetitive tasks — save time
  • Consistent, error-free operations
  • Run against hundreds of VMs/hosts
  • Schedule tasks (with Windows Task Scheduler)
  • Works across vCenter, NSX, vSAN, Horizon
  • Open-source community modules on PowerShell Gallery
Based on: Windows PowerShell / PowerShell Core
Cmdlet count: 700+ cmdlets
License: Free (from VMware)
PowerShell vs PowerCLI

💡
Exam Tip: "Compare PowerShell and PowerCLI" = table above + key point: PowerCLI extends PowerShell with VMware-specific cmdlets and a connection to vCenter via Connect-VIServer.
Installing PowerCLI — Windows, macOS, Linux
🔴 Must Know
PowerCLI is now distributed as a PowerShell module via the PowerShell Gallery — no separate installer needed. Requires PowerShell (Windows) or PowerShell Core (macOS/Linux).

Installation on Windows:

# Step 1: Install PowerShell (comes pre-installed on Windows) # Step 2: Install PowerCLI from PowerShell Gallery Install-Module -Name VMware.PowerCLI -Scope CurrentUser # Step 3: Trust the repository if prompted # Step 4: Verify installation Get-Module VMware.PowerCLI -ListAvailable

Installation on macOS / Linux:

# Step 1: Install PowerShell Core (open source) # macOS (via Homebrew): brew install --cask powershell # Linux (Ubuntu): sudo apt-get install -y powershell # Step 2: Launch PowerShell pwsh # Step 3: Install PowerCLI module (same as Windows) Install-Module -Name VMware.PowerCLI -Scope CurrentUser
💡
Exam Tip: Installation steps are directly asked. Remember: PowerCLI is installed as a module via Install-Module cmdlet. Same command works on all platforms once PowerShell/pwsh is installed.
Essential PowerCLI Commands
🔴 Must Know
# ── Connect to vCenter Server ── Connect-VIServer -Server "vcenter.lab.local" -User "[email protected]" -Password "P@ssw0rd" # ── Get all VMs ── Get-VM # ── Get a specific VM ── Get-VM -Name "WebServer01" # ── Power on / off a VM ── Start-VM -VM "WebServer01" Stop-VM -VM "WebServer01" -Confirm:$false # ── Create a new VM ── New-VM -Name "NewVM" -VMHost "esxi01.lab.local" -Datastore "DS01" -NumCPU 2 -MemoryGB 4 # ── Migrate (vMotion) a VM ── Move-VM -VM "WebServer01" -Destination (Get-VMHost "esxi02.lab.local") # ── Get all ESXi hosts ── Get-VMHost # ── Get all Datastores ── Get-Datastore # ── Create snapshot ── New-Snapshot -VM "WebServer01" -Name "Pre-Patch" -Memory -Quiesce # ── Disconnect from vCenter ── Disconnect-VIServer -Confirm:$false
💡
Exam Tip: Know Connect-VIServer, Get-VM, Start-VM, Stop-VM, New-VM, Move-VM, Get-VMHost, New-Snapshot. These are the most commonly asked PowerCLI commands. Pattern: Verb-Noun (Get, New, Set, Remove, Start, Stop, Move).
Naming convention: Verb-Noun (PowerShell standard)
Connect first: Always Connect-VIServer before any cmdlet
PowerCLI Scripting — Practical Examples
🟠 High Priority

Bulk operations — the real power of PowerCLI:

# Power on ALL VMs in a folder Get-VM -Location "Production" | Start-VM # Report CPU and Memory for all VMs Get-VM | Select-Object Name, NumCPU, MemoryGB, PowerState | Export-CSV "vm_report.csv" # Take snapshot of all powered-on VMs Get-VM | Where-Object {$_.PowerState -eq "PoweredOn"} | New-Snapshot -Name "Weekly-Backup" # Find VMs with snapshots older than 7 days Get-VM | Get-Snapshot | Where-Object {$_.Created -lt (Get-Date).AddDays(-7)}
