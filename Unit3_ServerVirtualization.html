Creating & Managing Virtual Machines

VM internals · externals · sizing · creation

VM Internals vs Externals
🔴 Must Know

VM Internals (logical components)

Components managed by VMware — exist as files on a datastore
FilePurpose
.vmxVM configuration file (text) — CPU, RAM, devices
.vmdkVirtual disk — stores VM's OS and data
.nvramBIOS/UEFI settings of the VM
.vmsdSnapshot database — tracks snapshot tree
.vmsnSnapshot state file — VM memory state
.logVM log file for troubleshooting
.vmssSuspend state (when VM is suspended)

VM Externals (virtual hardware)

Virtual hardware devices presented to the guest OS
  • vCPU — Virtual processors (sockets × cores)
  • vRAM — Virtual memory allocated to VM
  • vNIC — Virtual network adapter (VMXNET3, E1000e)
  • VMDK — Virtual hard disk (thin/thick provisioned)
  • CD/DVD — Virtual optical drive
  • USB controller — Virtual USB device support
  • Display adapter — SVGA virtual display
💡
Exam Tip: "Explain VM internals and externals" = files (.vmx, .vmdk, .nvram etc.) + virtual hardware (vCPU, vRAM, vNIC, VMDK). Most asked in 6-mark format.
Sizing Virtual Machines — CPU, Memory, Disk
🔴 Must Know
ResourceSizing GuidelinesKey Considerations
vCPUMatch to workload; start small (1–2 vCPUs) and growOver-provisioning vCPUs causes CPU scheduling contention; ratio: ≤4 vCPUs per pCPU recommended
vRAMAllocate what the workload needs; avoid excessOver-allocation triggers ballooning, swapping — degrades performance
vDisk (VMDK)Thin provision for flexibility; thick eager-zeroed for databasesThin: allocates on demand. Thick lazy: pre-allocates but no zeroing. Thick eager: pre-allocates + zeros — best I/O.

Thin Provisioning

Disk grows on demand up to max size. Saves space but can cause over-commitment issues.

Thick Eager Zeroed

Pre-allocates all space AND zeroes blocks at creation. Best performance — used for databases, FT VMs.

Thick Lazy Zeroed

Pre-allocates space but zeroes blocks on first write. Balance between speed of creation and performance.

Right-Sizing

Use vCenter performance charts to identify over/under-provisioned VMs and adjust accordingly.

💡
Exam Tip: Always remember: Thin = flexible but risky; Thick Eager Zeroed = best performance for production. FT VMs require Thick Eager Zeroed.
VM Snapshots
🟠 High Priority
Snapshot captures the state of a VM (memory, disk, VM settings) at a point in time. Allows reverting to a previous state. Not a backup — snapshots are delta files that grow over time.
/* Snapshot Delta Chain */ Base VMDK → Snapshot 1 delta → Snapshot 2 delta → Current state ↑ ↑ ↑ (original) (changes from (changes from base to S1) S1 to S2) Revert to Snapshot 1: discards all changes after S1 Delete snapshot: merges delta into parent → consolidates
Files created: .vmsd + .vmsn + delta .vmdk
Best practice: Delete snapshots after use; max 3 in chain
Impact: Disk I/O degrades with many snapshots
Naming Virtual Machines
🟢 Good to Know
VM naming in vSphere is the process of assigning a meaningful, consistent name to a virtual machine in the vCenter inventory. The VM name (inventory name) is separate from the guest OS hostname.

VM Inventory Name

  • Name shown in vCenter inventory — for admin identification
  • Up to 80 characters; can include spaces
  • Does NOT need to match guest OS hostname
  • Best practice: use a consistent naming convention (e.g., Role-OS-Env: WebSrv-Win2019-Prod)

Naming Convention Best Practices

  • Include: role (web, db, app), OS (win, rhel), environment (prod, dev, test)
  • Include: site/location code and sequence number (e.g., MUM-WEB-PROD-01)
  • Keep consistent across the organization
  • Avoid spaces in names used for scripting/automation
Max length: 80 characters
Guest hostname: Set separately in guest OS or Customization Spec

Cloning · Customization · Instant Clone · Creating Templates · Deploying from Templates · OVF/OVA

💡
Exam Tip: "Template vs Clone" = table above. Key points: Template cannot be powered on; used as gold image; Customization Spec changes hostname/IP/SID at deploy time.
Customization Specifications
🟠 High Priority
Customization Specification is a saved set of configuration settings applied to a VM when deployed from a template — automatically customizes the guest OS (hostname, IP, domain, SID) so each deployed VM is unique.
  • Prevents duplicate SID (Security Identifier) issues in Windows VMs
  • Sets unique hostname, IP address, DNS, time zone
  • Can run a custom script after deployment (post-deploy automation)
  • Saved in vCenter and reusable across multiple deployments
  • Requires VMware Tools to be installed in the guest OS
Requires: VMware Tools in guest OS
Prevents: Duplicate SID / IP conflicts
Creating VM Templates
🟠 High Priority
A template is a master copy (gold image) of a VM used for repeated, standardized deployments. Once converted to a template, the VM cannot be powered on — it only exists to be cloned/deployed from.

Steps to Create a Template:

1
Prepare the source VM — install OS, apply patches, install apps, run Sysprep (Windows) or waagent (Linux) to generalize
↓
2
Install VMware Tools — required for customization spec to work during deployment
↓
3
Power off the VM → Right-click → "Clone to Template" or "Convert to Template"
↓
4
Template stored in datastore — shown in VMs and Templates view in vSphere Client

Clone to Template

Creates a template copy while keeping the original VM intact — VM still exists and can be used

Convert to Template

Converts the VM itself into a template — original VM is gone; only template remains

💡
Exam Tip: Know the difference: "Clone to Template" keeps the VM; "Convert to Template" removes the VM. Templates are stored in the same datastore as VMs.
Deploying VMs from Templates
🟠 High Priority
Deploying from a template creates one or more new VMs from a template — each deployed VM gets its own identity (hostname, IP, SID) via a Customization Specification applied during deployment.

Deployment Steps:

1
Right-click the template → "New VM from This Template" (or "Deploy VM from Template")
↓
2
Wizard: select name, folder, compute resource (host/cluster), storage (datastore), disk format (thin/thick)
↓
3
Select Customization Specification — sets unique hostname, IP address, domain, SID for this VM
↓
4
Review summary → Finish → VM is created, customized, and powered on automatically
No customization spec: Deployed VM has same hostname/IP as template source — conflict!
Content Library: Store templates centrally; deploy across multiple vCenters
Instant Clone
🟠 High Priority
Instant Clone creates a new VM from a running parent VM — the clone shares the parent's memory pages (copy-on-write) and disk, enabling near-instantaneous VM creation (milliseconds).

How it works

  • Parent VM is quiesced momentarily
  • Fork creates child from parent's memory
  • Child gets its own identity (via customization)
  • Parent resumes immediately

Use cases

  • VMware Horizon VDI (virtual desktops)
  • Cloud-native app scaling
  • Rapid test environment provisioning
  • Stateless workloads that need fast spin-up
Speed: Milliseconds (shares parent memory)
vs Clone: Full clone = minutes; Instant = ms
OVF / OVA Templates
🔴 Must Know
OVF (Open Virtualization Format) is an open standard for packaging and distributing virtual appliances. An OVA (Open Virtual Appliance) is a single-file archive (.ova) of an OVF package.
FeatureOVFOVA
FormatMultiple files (.ovf descriptor + .vmdk + .mf manifest)Single .ova archive (tar of all OVF files)
PortabilityOpen standard — works across different hypervisorsSame — easier to distribute (one file)
Use caseSharing VMs/appliances between environmentsDeploying pre-built appliances (e.g., VCSA, NSX)

OVF Operations in vSphere:

Deploy OVF/OVA

Import a virtual appliance into vSphere from local disk or URL — wizard-based in vSphere Client

Export OVF/OVA

Export a VM or vApp to OVF/OVA format — preserves disk, config, and hardware version

Examine OVF

View OVF descriptor (.ovf file) to check VM requirements, disk sizes, properties before deploying

Content Library

vCenter repository for templates, OVF/OVAs, ISOs — enables sharing across multiple vCenter instances

💡
Exam Tip: OVF vs OVA = multi-file vs single-file. Key operations: Deploy, Export, Examine. Mention Content Library as vCenter's central template store.
vApps
🟢 Good to Know
vApp is a container for multiple related VMs that function together as a single application (e.g., a 3-tier web app: web server + app server + DB server). A vApp can be managed, started, stopped, and exported as a unit.
  • Has its own power-on/off sequence — start DB first, then app server, then web server
  • Can be exported as OVF/OVA (entire multi-VM app in one package)
  • Supports IP allocation policies and properties for VMs within
  • Displayed as a container in the vSphere inventory alongside VMs

Managing Resource Allocation

Shares · Reservations · Limits · Resource Pools · CPU · Memory

VM Resource Allocation — Shares, Reservation, Limit
🔴 Must Know
ControlDefinitionExampleWhen matters
SharesRelative weight during resource contention — more shares = higher priorityVM-A: 2000 shares; VM-B: 1000 shares → VM-A gets 2× CPU in contentionOnly during resource contention
ReservationGuaranteed minimum resources — always available, even under contentionReserve 2 GHz CPU → VM always gets at least 2 GHzMission-critical VMs that need guaranteed resources
LimitMaximum resources a VM can consume — caps usageLimit 4 GHz CPU → VM never uses more than 4 GHz even if idle resources availableCost chargeback, preventing resource hogging
/* Resource Allocation Analogy */ Shares → Like priority lanes: more shares = faster during congestion Reserve → Like a reserved seat: guaranteed, no matter what Limit → Like a speed limit: maximum allowed even if road is empty
💡
Exam Tip: This is a very common 6-mark question. Define all three clearly with an analogy or example. Key: Shares = relative priority; Reservation = guarantee; Limit = cap. Shares only matter during contention.
Resource Pools
🟠 High Priority
Resource Pool is a logical grouping of CPU and memory resources on an ESXi host or cluster. VMs placed in a resource pool share its allocated resources — allows hierarchical resource management.
/* Resource Pool Hierarchy Example */ Cluster (Total: 64 vCPU, 256 GB RAM) ├── Resource Pool: Production (High Shares, 100% Reserve) │ ├── VM: WebServer01 │ └── VM: DBServer01 ├── Resource Pool: Development (Low Shares, No Reserve) │ ├── VM: DevVM01 │ └── VM: TestVM01 └── Resource Pool: QA (Medium Shares) └── VM: QATestVM
  • Each pool has its own Shares, Reservation, and Limit settings
  • Pools can be nested — child pools inherit parent's available resources
  • Useful for multi-tenant environments and department-based resource management
ESXi Advanced Memory Technologies
🔴 Must Know
ESXi uses multiple techniques to manage physical memory efficiently when VMs collectively request more memory than physically available — in order of preference (least to most disruptive).
TechniqueHow It WorksPerformance Impact
Transparent Page Sharing (TPS)Identical memory pages across VMs are deduplicated — shared read-only, copied on write (CoW). Saves memory.Minimal — no performance hit
Memory BallooningBalloon driver (in VMware Tools) inflates inside the guest — forces guest OS to reclaim memory from its own apps. ESXi reclaims freed pages.Moderate — guest experiences memory pressure
Memory CompressionBefore swapping, ESXi compresses pages into a compression cache — faster to decompress than swap from disk.Moderate — CPU used for compression
Host-Level SwappingESXi swaps VM memory pages to a swap file on disk (.vswp) when all other techniques exhausted.Severe — disk I/O is orders of magnitude slower than RAM
/* Memory Reclamation Order (ESXi Priority) */ 1. TPS (Transparent Page Sharing) ← no penalty 2. Ballooning ← minor penalty 3. Compression ← moderate penalty 4. Swap to disk (.vswp) ← severe penalty ⚠️
💡
Exam Tip: This is THE most asked memory question in this unit. Must know all 4 techniques in order. Remember: Ballooning requires VMware Tools. Swapping is worst-case scenario.
Memory Allocation Control
🟠 High Priority
Memory Allocation Control refers to the mechanisms ESXi administrators use to manage how physical RAM is distributed across VMs — using Reservations, Limits, Shares, and memory overcommitment policies.
Control MechanismEffect on MemoryUse Case
ReservationGuarantees a minimum amount of physical RAM always available to the VM — cannot be reclaimed by ESXiMission-critical VMs (DB servers, real-time apps)
LimitCaps maximum memory the VM can use — even if more physical RAM is freeDev/test VMs; prevent memory hogging
SharesRelative priority during memory contention — High (20 shares/MB), Normal (10), Low (5)Prioritize production VMs over dev VMs during pressure
Memory OvercommitmentTotal configured vRAM across VMs exceeds physical RAM — ESXi uses TPS/Balloon/Compression/Swap to manageEnvironments where VMs don't all use max RAM simultaneously

Swap File (.vswp)

  • Created per VM when it powers on
  • Size = configured vRAM minus reservation
  • Located on VM's datastore by default
  • Used only as last resort memory reclamation

Memory Hot-Add

  • Add vRAM to a running VM without power cycle
  • Must be enabled in VM settings before power-on
  • Guest OS must support memory hot-add
  • Useful for scaling without downtime
High shares: 20 per MB
Normal shares: 10 per MB
Low shares: 5 per MB
CPU Resource Management
🟠 High Priority
Default CPU Allocation: Each VM gets equal shares of CPU by default (1000 shares per vCPU). ESXi's CPU scheduler manages allocation using a proportional share mechanism.

CPU Ready

Time a VM waits for a physical CPU to become available. High CPU Ready = over-provisioned host. Acceptable: <10%

CPU Co-Stop

Time SMP VMs wait for all vCPUs to be scheduled simultaneously. Issue for VMs with many vCPUs on busy hosts.

Hyper-Threading

ESXi is aware of HT — treats logical cores (threads) separately. VMs can use HT cores but performance ≠ physical core.

NUMA Awareness

ESXi scheduler is NUMA-aware — tries to keep VM's vCPUs and memory on the same NUMA node for best performance.

Default shares: 1000 per vCPU
High shares: 2000 per vCPU
Low shares: 500 per vCPU
