vSphere Update Manager (VUM)

Patching, upgrading, and managing ESXi hosts and VMs

What is VUM?
🔴 Must Know
VUM (vSphere Update Manager) is a VMware tool integrated with vCenter Server that automates patching and upgrading of ESXi hosts, VMware Tools, and VM hardware versions. It ensures the environment stays compliant, secure, and up to date.
FeatureDescription
BaselineA collection of patches/upgrades VUM uses to check compliance
ScanChecks hosts/VMs against baselines to find what's missing
StageDownloads patches to the host before remediation (reduces downtime)
RemediateApplies patches; host enters maintenance mode → reboots → exits maintenance mode
ComplianceCompliant / Non-Compliant / Unknown — status after scan
/* VUM Remediation Workflow */ Attach Baseline → Scan → Check Compliance ↓ Stage Patches (optional, pre-download) ↓ Remediate → Host enters Maintenance Mode ↓ Patches Applied → Host Reboots ↓ Host exits Maintenance Mode → Compliant ✓
💡
Exam Tip: Know the 4 key operations: Scan → Stage → Remediate → Compliance check. Also mention: VUM is integrated into vCenter from vSphere 6.5+; no separate install needed in 6.7.
VUM in 6.7: Integrated into vCenter (VCSA)
Remediation: Requires Maintenance Mode
VUM Baselines & Baseline Groups
🟠 High Priority

Patch Baseline

Contains a set of patches for ESXi hosts. Can be dynamic (auto-update) or fixed.

Upgrade Baseline

Used to upgrade ESXi host version (e.g., 6.5 → 6.7) or VM hardware version.

Extension Baseline

Adds third-party software or drivers (extensions) to ESXi hosts.

Baseline Group

Combines multiple baselines (patch + upgrade + extension) into one group for unified remediation.

vSphere Networking

Standard Switches · VLANs · NIC Teaming · VMkernel · TCP/IP Stacks

Virtual Switch (vSwitch) vs Physical Switch
🔴 Must Know
Virtual Switch (vSwitch) is a software-based layer-2 switch in ESXi that connects VMs to each other and to physical networks. It forwards traffic based on MAC addresses — just like a physical switch, but runs in software.
FeatureVirtual Switch (vSwitch)Physical Switch
LocationInside ESXi hypervisor (software)Physical hardware device
ManagementvSphere Client / CLICLI / dedicated management interface
Port typesVM Port Groups, VMkernel ports, UplinksAccess, Trunk, LACP ports
VLAN supportYes (VLAN tagging on port groups)Yes (native)
Spanning TreeNot needed (no loops possible)Required (STP/RSTP)
Traffic visibilityPromiscuous mode for monitoringPort mirroring / SPAN
CostZero (software)Hardware cost
💡
Exam Tip: "Compare virtual and physical switch" is a common 6-mark question. Use the table above — cover at least 5 differences. Always mention: vSwitch has no spanning tree because loops are impossible in a software switch.

VMkernel Networking & TCP/IP Stacks
🟠 High Priority
VMkernel port carries ESXi host's own network traffic (not VM traffic). Each VMkernel port is assigned a specific service.
VMkernel ServicePurposeRequired for
ManagementESXi host management trafficvCenter communication, SSH, DCUI
vMotionLive VM migration trafficvMotion (live migration)
iSCSIStorage traffic over IP networkSoftware iSCSI initiator
Fault ToleranceFT logging traffic between primary and secondary VMsvSphere FT
vSANvSAN cluster communicationVMware vSAN
NFC (Provisioning)VM cloning, snapshot, migrationStorage vMotion, cloning
TCP/IP Stacks: vSphere 6.x introduced multiple TCP/IP stacks, allowing different traffic types to use separate routing tables, DNS, and gateways.

• Default Stack — Management, vMotion (default)
• vMotion Stack — Dedicated stack for vMotion traffic only
• Provisioning Stack — NFC traffic (cloning, snapshots)
• Custom Stacks — For third-party solutions
💡
Exam Tip: TCP/IP Stacks is a unique vSphere 6.x feature — mention it to score extra. Key point: each stack has its own routing table and DNS, allowing isolation of traffic types.
VLANs in vSphere
🟠 High Priority
VLAN (Virtual LAN) segments network traffic within a vSwitch, providing isolation between groups of VMs without needing separate physical switches.
VLAN ModeHow it WorksVLAN Tag on Port Group
No VLAN (0)All traffic passes untagged — VMs share one broadcast domain0 (none)
VLAN (1–4094)ESXi tags/untags frames — VMs use specific VLAN1–4094
VLAN Trunking (4095)All VLANs pass through — VM OS handles tagging (for routers, firewalls in VMs)4095
💡
Exam Tip: Remember VLAN ID 4095 = trunk mode (all VLANs). This is tested as a tricky MCQ or short answer.
NIC Teaming & Load Balancing
🟠 High Priority
NIC Teaming = connecting multiple physical NICs (uplinks) to a single vSwitch for redundancy and/or load balancing. If one NIC fails, traffic automatically switches to the other.
Load Balancing PolicyHow It Works
Route based on originating port IDDefault — each VM uses one uplink based on its virtual port (simple, no switch config needed)
Route based on source MAC hashVM's MAC address determines which uplink — consistent per VM
Route based on IP hashSource+dest IP determines uplink — requires LACP/802.3ad on physical switch
Use explicit failover orderActive/standby: primary uplink always used, secondary only on failure
Default policy: Port ID
Needs physical switch config: IP Hash only
Multicast in vSphere Networking
🟢 Good to Know
Multicast sends network traffic from one source to multiple destinations simultaneously — more efficient than unicast (one-to-one) or broadcast (one-to-all).
  • Used in vSphere for: HA heartbeat communication, vSAN cluster discovery, and certain cluster services
  • vSphere HA uses multicast (224.x.x.x) or unicast for master/slave election
  • Requires multicast-capable physical switches for some features

Shared Storage in vSphere
🔴 Must Know
Shared storage allows multiple ESXi hosts to access the same datastore simultaneously — essential for features like vMotion, HA, and DRS. Without shared storage, VMs cannot be migrated between hosts.
Storage TypeProtocolKey Characteristics
Fibre Channel (FC) SANFibre ChannelHigh performance, dedicated storage network, uses HBA (Host Bus Adapter), expensive
iSCSI SANiSCSI (TCP/IP)SAN over IP network, lower cost than FC, software or hardware initiator
NFS (NAS)NFS (TCP/IP)File-level access, simple setup, shared over IP, flexible — no VMFS needed
FCoEFibre Channel over EthernetCombines FC and Ethernet — reduces cabling, uses CNA (Converged Network Adapter)
vSANVMware proprietaryHyperconverged — uses local disks of ESXi hosts to create shared storage pool
VMFS (Local/SAN)VMware FSVMware's clustered filesystem for SAN datastores — allows multiple hosts to mount same LUN

RAID in vSphere Storage
🟠 High Priority
RAID (Redundant Array of Independent Disks) combines multiple disks for performance and/or redundancy. Used in storage arrays connected to ESXi hosts.
RAID LevelNameMin DisksRedundancyPerformanceUse Case
RAID 0Striping2None ❌Best read/writeTemp data, not production
RAID 1Mirroring21 disk failureGood readOS drives, critical data
RAID 5Striping + Parity31 disk failureGood readGeneral purpose — most common
RAID 6Double Parity42 disk failuresModerateCritical storage, high availability
RAID 10Stripe of Mirrors41 per mirror pairBestDatabases, high I/O workloads
💡
Exam Tip: RAID 5 = most commonly used in production. RAID 10 = best performance + redundancy. RAID 0 = no redundancy (dangerous for production).
VMware vSAN (Virtual SAN)
🔴 Must Know
vSAN is VMware's hyperconverged storage solution that aggregates local disks from ESXi hosts in a cluster into a shared datastore — eliminating the need for external SAN/NAS hardware.
/* vSAN Architecture */ ESXi Host 1 ESXi Host 2 ESXi Host 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ SSD+HDD │ │ SSD+HDD │ │ SSD+HDD │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ ─────┴─────────────────┴─────────────────┴───── vSAN Network (10 GbE) ───────────────────────────────────────────────── vSAN Datastore (shared pool) ← All hosts see this as ONE datastore →

Requirements

  • Minimum 3 ESXi hosts
  • Each host: SSD (cache) + HDD/SSD (capacity)
  • 10 GbE network recommended
  • vSAN license required

Key Benefits

  • No external storage hardware
  • Storage grows by adding hosts
  • Policy-based management (SPBM)
  • Integrates natively with vSphere
Min hosts: 3
Disk groups: 1 SSD cache + 1–7 capacity disks
VMkernel port: Dedicated vSAN port needed

High Availability & Business Continuity

HA Layers · VM Clustering · NLB · WSFC

HA Layers in vSphere
🟠 High Priority
High Availability in vSphere operates at multiple layers — each layer provides a different level of protection and complements the others.
LayerMechanismProtects Against
Infrastructure HARedundant power supplies, network switches, storage paths (multipathing)Physical hardware failures in data center
vSphere HAAutomatically restarts VMs on surviving hosts after ESXi host failureESXi host failure — VM-level restart in minutes
vSphere FTShadow VM on secondary host via vLockstep — instant failoverESXi host failure — zero VM downtime
Application HA (WSFC/NLB)Windows clustering or load balancing within guest OSApplication failures and load distribution
Backup & ReplicationVM snapshots, vSphere Replication, third-party backup toolsData loss — recovery from backup
💡
Exam Tip: "Explain HA layers" = 5-row table above. Emphasize that each layer protects against different failure types — they are complementary, not alternatives.
VM Clustering in vSphere
🟠 High Priority
VM Clustering (also called Guest OS Clustering) refers to configuring multiple VMs to work together as a cluster at the guest OS level — providing application-level high availability independent of vSphere HA.

Cluster-in-a-Box (CIB)

  • Both VMs on the same ESXi host
  • Share a VMDK as shared disk
  • Protects against app failure only (not host failure)
  • Use only for testing / dev environments

Cluster-Across-Boxes (CAB)

  • VMs on different ESXi hosts
  • Shared RDM (Raw Device Mapping) disk on SAN
  • Protects against both app AND host failure
  • Recommended for production WSFC clusters
Shared disk options: RDM (physical mode) or Clustered VMDK
Examples: SQL Server Always On, Windows WSFC
vSphere High Availability (HA)
🔴 Must Know
vSphere HA automatically restarts VMs on surviving hosts when an ESXi host fails. It detects host failures and responds without manual intervention — minimizing downtime for mission-critical workloads.
HA ConceptExplanation
Master HostElected host that monitors slave hosts and manages VM restarts; elected by majority vote
Slave HostAll other hosts in the HA cluster; send heartbeats to master
HeartbeatPeriodic signal (every 1 second) sent between master and slaves to detect failures
Datastore HeartbeatSecondary mechanism using shared datastore to distinguish host failure from network partition
Admission ControlReserves cluster resources to ensure failover capacity is always available
Isolation ResponseWhat a host does when it can't reach master: Power off / Shut down / Leave VMs running
VM Restart PriorityHigh / Medium / Low — order in which VMs are restarted after failover
/* HA Failure Detection & Failover Flow */ Host A fails (no heartbeat for 15 sec) ↓ Master checks Datastore Heartbeat ↓ If no datastore heartbeat either → Confirmed failure ↓ Master restarts VMs from Host A on remaining hosts (in priority order: High → Medium → Low)
💡
Exam Tip: Draw the HA flow: Host failure → Master detects → Datastore heartbeat check → VM restart. Mention: master/slave election, admission control, isolation response. This is the most asked HA question.

Network Load Balancing (NLB) & Windows Server Failover Clustering (WSFC)
🟠 High Priority
NLB (Network Load Balancing) distributes incoming network traffic across multiple servers to prevent any single server from being overwhelmed — improves performance and availability for stateless workloads (web servers, etc.).

NLB in vSphere

  • VMs running NLB must be on same network segment
  • vSwitch must be set to Promiscuous Mode
  • Works at Layer 4 (TCP/UDP)
  • Suitable for stateless: web, DNS, FTP

WSFC in vSphere

  • Windows Server Failover Clustering — for stateful applications (SQL, Exchange)
  • Requires shared RDM (Raw Device Mapping) or Cluster-in-a-Box
  • VMs in same cluster can be on different ESXi hosts
  • Used for active/passive clustering

Securing VMware vSphere

ESXi Security · Authentication · Permissions · Patching · Logging · Boot Security

ESXi Security Best Practices
🔴 Must Know

Access Control

  • Disable SSH and ESXi Shell by default
  • Use DCUI for local management only
  • Restrict root account — create named admin accounts
  • Use Host Client (HTTPS) for remote access

Authentication

  • Integrate ESXi with Active Directory (AD)
  • Use vCenter SSO for central auth
  • Enable two-factor authentication (2FA)
  • Configure lockdown mode (Normal / Strict)

Patching & Updates

  • Use VUM for systematic patch management
  • Keep ESXi and VMware Tools up to date
  • Apply security patches promptly
  • Use baselines for compliance tracking

Logging & Audit

  • Configure syslog to remote syslog server
  • Enable audit logging in vCenter
  • Monitor /var/log on ESXi for security events
  • Use vRealize Log Insight for centralized logging
Permissions & Role-Based Access Control (RBAC)
🔴 Must Know
vSphere Permissions = Role + User/Group + Object. A permission grants a user (or group) a specific role on a specific vSphere object (e.g., a VM, host, datacenter).
ComponentDescription
RoleA named collection of privileges (e.g., Administrator, Read-Only, VM Power User)
PrivilegeA single granular permission (e.g., "VM.PowerOn", "Datastore.Browse")
ObjectThe vSphere inventory item the permission applies to (VM, host, cluster, datacenter)
PropagationPermissions set on a parent object (datacenter) automatically apply to child objects (hosts, VMs)
Global PermissionsApply across entire vCenter inventory (top-level); supersede object-level permissions
/* Built-in Roles in vSphere */ Administrator ← Full access to all objects Read-Only ← View only, no changes No Access ← Cannot see or interact with the object VM Power User ← Interact with VMs but not configure Datastore Consumer ← Allocate space on datastores
💡
Exam Tip: "Explain RBAC in vSphere" = define Role + Privilege + Object + Permission propagation. Mention built-in roles. Always state: permissions propagate from parent to child by default.
ESXi Lockdown Mode
🟠 High Priority
Lockdown Mode restricts ESXi host access so that management is only possible through vCenter Server — direct host login is blocked. This hardens security by centralizing control.
ModeSSH/ShellDCUIvCenter APIException Users
NormalDisabledAllowedAllowedException user list respected
StrictDisabledDisabledAllowedOnly exception users via vCenter
ESXi Boot Process Security
🟢 Good to Know
  • Secure Boot — UEFI feature that verifies ESXi bootloader and kernel using cryptographic signatures; prevents loading of unauthorized/tampered code
  • TPM 2.0 — Trusted Platform Module; stores cryptographic keys; used for VM Encryption key storage and attestation
  • vTPM — Virtual TPM added to VMs; allows guest OS to use TPM-based features (BitLocker, Windows Hello)
  • Encrypted Boot — ESXi configuration can be encrypted; requires TPM to decrypt at boot
